# Apple/macOS 勒索软件深度分析(约 2021–2026)

本文聚焦近 5 年 Apple 平台(macOS / Apple Silicon)勒索威胁,从**威胁现状、代表样本、
技术演进、防御体系、检测难点**展开,并落到本工程 PMU 检测的适配。

> ⚠️ 重要前提:① 内容为公开威胁情报与体系结构知识的归纳,**不含攻击实现细节**;
> ② 作者知识存在时效性,**2025–2026 的最新个案可能不完整**,文中对近两年以"趋势研判"
> 表述、不臆造具体样本名;③ 数字/结论以公开报告口径为准,落地前请用最新情报复核。

## 一、总体判断:稀少但战略价值上升

- **绝对量仍少**:相比 Windows,真正在野、成规模加密 Mac 用户文件的勒索家族**非常少**。
  多数"Mac 勒索"新闻实为:**概念验证(PoC)、生成器(builder)、跨平台 RaaS 的 macOS
  分支测试样本**,而非大规模在野活动。
- **战略价值高**:Mac 在**企业、开发者、高净值人群**渗透率上升,**开发者机器 = 供应链
  入口**(Xcode、CI、签名证书、依赖仓库),使 Apple 成为**定向攻击**的高价值目标。
- **结论**:威胁形态从"广撒网勒索"转向"**定向 + 供应链 + 数据外泄勒索(双重勒索)**",
  纯加密勒索反而不是 Mac 上的主要形态。

## 二、代表性历史样本(奠定认知基线)

| 样本 | 时间 | 性质 | 关键特征 |
|------|------|------|---------|
| **KeRanger** | 2016 | 首个"功能完整"的 macOS 勒索 | 经被篡改的 Transmission(带合法开发者签名)分发,延时加密 |
| **EvilQuest / ThiefQuest** | 2020 | 勒索 + 窃密混合体 | 伪装盗版软件,兼带键盘记录/窃取文件,加密只是其一 |
| **(各 RaaS 的 macOS 分支)** | 2022– | 跨平台家族试探 | LockBit 等放出过 macOS/Apple Silicon 测试样本,**成熟度低、多被证实为半成品** |

**认知基线**:Mac 勒索史上的"标志事件"屈指可数,且常**夹带窃密**(EvilQuest)或**依附
合法签名分发**(KeRanger)。这两点延续至今,是 Mac 平台的典型打法。

## 三、近 5 年技术演进趋势(2021–2026)

### 1. 从"纯加密"转向"窃密优先 + 双重勒索"
Apple 的恢复机制(Time Machine、iCloud、APFS 快照)使纯加密的"赎金杠杆"变弱,
攻击者更倾向**先外泄数据再威胁公开**——数据外泄成为主杠杆,加密退居其次。

### 2. 瞄准供应链与开发者链路
- **Xcode/依赖投毒**(如历史上的 XcodeGhost 思路延续)、CI 凭据窃取、签名证书盗用;
- 开发者机器一旦失陷,可波及其**分发的下游软件**,杠杆远超单机加密。

### 3. 跨平台语言与 RaaS 化
- 用 **Go / Rust** 编写以便一套代码覆盖 Win/Linux/macOS(BlackCat/ALPHV 是跨平台典型);
- Apple Silicon(arm64)成为必须适配的目标架构,出现 **universal/arm64 原生**样本试探。

### 4. 滥用合法工具与"自带签名"
- 借**合法签名/公证**或**被盗证书**绕过 Gatekeeper;
- 滥用系统自带二进制(LotL)、合法加密/压缩工具完成部分操作以降低特异性。

### 5. 社工诱导授权(对抗 TCC)
- 因 TCC 限制对"文档/桌面/下载"等目录的访问,样本常**诱导用户主动授权**或
  伪装成需要权限的合法应用。

## 四、Apple 的多层防御如何抬高门槛

macOS 勒索"难成规模"很大程度归功于系统防御:

| 机制 | 作用 | 对勒索的阻断点 |
|------|------|--------------|
| **Gatekeeper + 公证(Notarization)** | 拦截未签名/未公证程序 | 阻止任意载荷直接执行 |
| **SIP(系统完整性保护)** | 锁系统目录,即便 root 也不能改 | 阻止破坏系统级恢复 |
| **TCC(隐私权限)** | 访问用户文档/相册/外置卷需授权 | **阻止静默全盘加密**(关键) |
| **App Sandbox** | 限制应用可触达资源 | 缩小加密可达范围 |
| **签名系统卷(SSV)** | 系统卷只读且密封校验 | 系统分区无法被加密篡改 |
| **XProtect / XProtect Remediator** | 内置特征/行为查杀 | 拦已知家族、做行为补救 |
| **Time Machine / APFS 快照** | 快速恢复 | 削弱赎金杠杆 |

→ **静默、全自动、全盘加密**在现代 macOS 上**很难实现**;攻击者被迫走"社工授权 +
有限范围加密 + 数据外泄"路线。

## 五、Apple Silicon 的硬件检测难点(本工程视角)

把本工程的 PMU 检测迁到 Apple Silicon,有三个**特有障碍**:

### 1. PMU 是 Apple 私有实现
- Apple Silicon 是 ARMv8/v9,但 **PMU 事件编号与语义不遵循 ARM 标准**;
- 通过私有框架 **`kpc`/`kperf`** 或 **Instruments / `powermetrics`** 访问,
  接口不稳定、文档少、权限高;
- 本工程的 `crypto_spec`(架构事件 0x77)**不能直接套用**,需重新映射到 Apple 事件。

### 2. 硬件 AES 引擎旁路指令计数
- Apple Silicon 有**专用硬件加密引擎/协处理器**,AES 等可能**不走通用指令**,
  因而**不体现在"加密指令占比"上**——`crypto_ratio` 信号在 Mac 上可能显著减弱;
- **应对**:更依赖**缓存遍历(LLC/L1 未命中)+ 写突发 + 突发节奏(差分)**类特征,
  这些不依赖加密指令计数,恰是本工程 12 档与 `d_*` 差分特征的强项。

### 3. 统一内存架构(UMA)改变访存特征
- Apple Silicon 的 CPU/GPU 共享统一内存,缓存层级与未命中行为**与标准 ARM/x86 不同**,
  阈值与基线需在 Apple 硬件上**重新标定**,不能照搬本工程合成数据的分布。

## 六、对本工程的适配清单(若迁移到 macOS)

1. **事件映射**:用 Apple PMU 实际事件替换 13 个 ARMv8 计数器,优先保住缓存类与
   指令构成类(`l1d/ll_cache` 对应项、`inst_retired`、`inst_spec`);
2. **弱化 crypto 依赖**:鉴于硬件 AES 引擎,降低对 `crypto_ratio` 的权重,
   提升缓存遍历与节奏特征;
3. **重标基线**:在真实 Apple Silicon 上采集良性/勒索样本,重训而非套用合成分布;
4. **补足非 PMU 信号**(Mac 上尤其重要):
   - **TCC 授权事件**(异常的全量文档访问授权请求);
   - **Time Machine/APFS 快照删除**(类比 Windows 删卷影,是强预警);
   - **文件熵 + 批量改扩展名**;
   - **签名/公证状态 + Endpoint Security 框架(ESF)事件**(`ES_EVENT_TYPE_*`)。
5. **采集层**:用 **Endpoint Security Framework** 拿文件/进程事件,PMU 经 `kpc`,
   二者融合。

## 七、结论

- **现状**:Mac 勒索近 5 年**在野规模仍小**,但形态升级为"**定向 + 供应链 + 双重勒索**",
  开发者与企业 Mac 是高价值目标;
- **系统防御**(Gatekeeper/SIP/TCC/SSV/快照)是 Mac 勒索难成规模的主因,也把攻击逼向
  社工授权与数据外泄;
- **硬件检测**在 Apple Silicon 面临 **PMU 私有 + 硬件 AES 旁路 + 统一内存**三重难点,
  迁移本工程需**换事件映射、弱化 crypto 依赖、重标基线、强配 ESF/文件/签名信号**;
- **本工程定位**:在 Mac 上,PMU 更适合作为"**缓存遍历 + 节奏**"的辅助硬件信号,
  与 Endpoint Security 框架的文件/进程/授权事件融合,而非单打独斗。
